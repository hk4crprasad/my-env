The Problem

LLMs Are Bad at Email Triage

Zero-shot failures

  • ❌ Routes billing disputes → Engineering (60%)
  • ❌ Classifies phishing as "urgent"
  • ❌ Misses escalation on GDPR threats
  • ❌ Responds to spam with helpful links
  • ❌ Ignores thread context entirely

Why RL, not classification?

A classifier picks one label.

Our agent makes 5 coordinated decisions per email — and each is verified independently.

No single correct trace to imitate → RL is the right tool.

Training Strategy

3-Level Curriculum Learning

Cold-start on hard → near-zero reward → training stalls.
Curriculum bootstraps the policy, then advances automatically.

1

Easy — 5 emails, 1 dimension, clear signals

Classification only. Clear spam/billing/technical/urgent/general. Model gets non-zero reward immediately. Advances when avg_reward > 0.60

2

Medium — 10 emails, 3 dimensions, phishing + hybrids

Adds priority + routing. Phishing disguised as urgent alerts. Billing-urgent hybrids. Advances when avg_reward > 0.50

3

Hard — 20 emails, 5 dimensions, thread context

All 5 dimensions. Reply chains needing context. GDPR audits. XSS disclosures. Response drafts required. Red herrings throughout.

Full Pipeline

How GRPO Training Works

Environment handles world dynamics · GRPO handles optimisation · Model just learns to act

📧

Deterministic Email Corpus

Seed-based generation — same seed → identical inbox → reproducible scores

🔁

Model Generates 4 Completions per Prompt

GRPO samples N=4 actions per email. More diversity → better gradient signal.

⚖️

7 Independent Verifiers Score Each

No learned reward model needed — pure deterministic functions. Each fires independently.

📈

GRPO Shifts Probability Mass

Higher-reward completions become more likely. No value model — advantage estimated from group.

Training Stack

  • 🤗 TRL GRPOTrainer — RL algorithm

  • 🌐 OpenEnv — environment interface
